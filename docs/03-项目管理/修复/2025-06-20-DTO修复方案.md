# DTO 修复方案

> 生成日期: 2025-06-20  
> 目标: 解决 DTO 定义未使用、API 响应暴露 Entity 的架构问题

---

## 问题分析

### 当前问题

| 层级 | 问题 | 影响 |
|------|------|------|
| Application Service | 返回 `Entity` 而非 `DTO` | 暴露内部结构，导航属性序列化问题 |
| Controller | 接收/返回 `Entity` 而非 `DTO` | API 响应包含循环引用风险 |
| DTO | 已定义但未使用 | 无实际作用 |

### 问题代码示例

```csharp
// Application Service 当前返回 Entity
public async Task<IEnumerable<WorkOrder>> GetAllAsync()
{
    return await _workOrderRepo.GetAllAsync();  // ❌ 返回 Entity
}

// Controller 当前接收/返回 Entity
public async Task<IActionResult> GetAll()
{
    var result = await _service.GetAllAsync();
    return Success(result);  // ❌ 直接返回 Entity
}
```

### DTO 使用示例

```csharp
// Application Service 修复后返回 DTO
public async Task<IEnumerable<WorkOrderDto>> GetAllAsync()
{
    var entities = await _workOrderRepo.GetAllAsync();
    return entities.Select(MapToDto);  // ✅ 返回 DTO
}

// Controller 修复后返回 DTO
public async Task<IActionResult> GetAll()
{
    var result = await _service.GetAllAsync();
    return Success(result);  // ✅ 返回 DTO
}
```

## 修复方案

### 方案对比

| 方案 | Application Service | Controller 请求 | Controller 响应 | 工作量 |
|------|---------------------|-----------------|-----------------|--------|
| A: 完全 DTO | 返回 DTO | 接收 DTO | 返回 DTO | 较大 |
| **B: 输出 DTO（推荐）** | 返回 DTO | 接收 Entity | 返回 DTO | 中等 |
| C: 输入 DTO | 返回 Entity | 接收 DTO | 返回 Entity | 较小 |

### 推荐方案 B

**理由**：
- Controller 接收 Entity 可以复用模型验证（`[Required]` 等 DataAnnotation）
- 主要问题是 API 响应暴露 Entity（循环引用、导航属性）
- 工作量适中

## 修复后的架构

```
Layer Flow (修复后):

API 请求
    ↓
Controller (接收 Entity，复用验证)
    ↓
Application Service (调用领域逻辑)
    ↓
Repository (返回 Entity)
    ↓
Application Service 映射 Entity → DTO
    ↓
Controller 返回 DTO
    ↓
API 响应 (干净的 DTO，无循环引用)
```
