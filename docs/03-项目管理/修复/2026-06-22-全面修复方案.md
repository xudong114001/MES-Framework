## 二、详细问题及修复方案

### P1: 架构违规修复

#### 1.1 Controller 注入 Domain 仓储 → 改为通过 Application Service

**问题**：多个 Controller 直接注入了 `IRepository<T>`，违反 Clean Architecture 依赖规则。

**涉及文件**：
- `src/MES.Api/Controllers/QcController.cs` — 注入 `IRepository<QcInspection>` 和 `IRepository<QcInspectionItem>`
- `src/MES.Api/Controllers/EquipmentController.cs` — 注入 `IRepository<Equipment>`

**修复方案**：
- 将这些仓储调用迁移到对应的 Application Service 中
- Controller 只注入 Service 接口，不再直接操作仓储

#### 1.2 Controller 直接接收 Entity → 改为接收 DTO

**问题**：Controller 直接接收 Entity 作为请求体参数，违反 DTO 映射规范。

**涉及文件**：
- `EquipmentController.Create([FromBody] Equipment entity)`
- `EquipmentController.Update(long id, [FromBody] Equipment entity)`
- `WorkReportController.Submit([FromBody] WorkReport report)`
- `WorkReportController.Update(long id, [FromBody] WorkReport entity)`

**修复方案**：
- 创建对应的 DTO 类（如 `CreateEquipmentRequest`、`UpdateEquipmentRequest`）
- Controller 改为接收 DTO，Service 内部映射到 Entity

#### 1.3 Controller try-catch → 由 ExceptionMiddleware 统一处理

**问题**：规范要求"Controller 不写 try-catch，异常由 ExceptionMiddleware 统一处理"，但多个 Controller 仍手动捕获异常。

**涉及文件**：
- `QcController.cs` — CreateInspection、AddItem、VerifyInspection、HandleNonconforming
- `RoleController.cs` — AssignPermissions、Delete
- `AndonController.cs` — 需检查
- `SchedulingController.cs` — 需检查
- `QcCheckpointController.cs` — 需检查
- `DispatchController.cs` — 需检查
- `IntegrationController.cs` — 需检查
- `SeedController.cs` — 需检查

**修复方案**：
- 移除所有 Controller 中的 try-catch
- 修复 ExceptionMiddleware 使其能正确分发不同类型的异常

#### 1.4 ExceptionMiddleware 区分异常类型

**问题**：当前 ExceptionMiddleware 将所有异常统一返回 500。

**修复方案**：
```csharp
catch (DomainException ex)      → HTTP 400
catch (EntityNotFoundException) → HTTP 404
catch (ForbiddenException)      → HTTP 403
catch (ValidationException)     → HTTP 400（含字段级错误）
catch (Exception)               → HTTP 500（仅记录日志）
```

### P2: 代��质量��复

#### 2.1 Service 接口返回类型弱化

**问题**：`IRoleService` 等接口的方法返回 `object` 或 `IEnumerable<object>`，丧失类型安全。

**修复方案**：
- 定义具体的 `RoleDto` 类
- 修改接口签名使用强类型 DTO

#### 2.2 缺少聚合根标记

**问题**：CLAUDE.md 列出了 9 个聚合根，但以下未实现 IAggregateRoot：
- Factory
- Material  
- WorkReport
- AndonEvent

**修复方案**：为上述实体添加 `: IAggregateRoot` 标记。

#### 2.3 Fire-and-forget 推送 SignalR

**问题**：WorkReportController 使用 `_ = Task.Run(async () => ...)` 推送 SignalR 通知，在 ASP.NET Core 中可能导致 HttpContext 提前释放或未捕获异常。

**修复方案**：
- 改为注入 `IHubContext<MesHub>` 到 Service 层，在 Service 中同步推送
- 或使用后台服务队列处理推送

#### 2.4 Role 实体贫血模型

**问题**：Role 实体只有属性没有业务方法。

**修复方案**：
- 为 Role 添加工厂方法如 `Role.Create(name, description)`
- 将 Service 中简单业务逻辑下沉到 Domain 层

### P3: 配置/部署修复

#### 3.1 环境变量文件对齐

**修复方案**：
- 统一 .env 和 .env.example 的字段和格式
- .env 中添加 JWT_SECRET_KEY

#### 3.2 nginx 配置添加前端

**修复方案**：
- 添加前端静态文件路径
- 配置 API 反向代理

#### 3.3 Dockerfile 清理

**修复方案**：
- 移除不必要的 MES.Tests.csproj 引用（构建 Api 不需要测试项目）

#### 3.4 Program.cs 重复注册

**问题**：WorkOrderService 注册了两次（L122 和 L123）

**修复方案**：移除重复注册行。
