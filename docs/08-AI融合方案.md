> 版本：1.0
> 日期：2026-06-06
> 状态：待评审

---

## 1. 背景与目标

### 1.1 背景

当前 MES-Framework 已具备完整的工单、报工、质检、设备、追溯等核心模块，积累了真实的生产数据（工单、报工记录、质检结果、设备 OEE）。在此基础上引入 AI 能力，可将 MES 从"记录系统"升级为"决策支持系统"。

### 1.2 目标

| 目标 | 说明 |
|------|------|
| 质量预警 | 从"事后发现废品"变为"事中预警干预" |
| 智能排程 | 降低调度员经验依赖，减少交期延误 |
| 设备预测 | 从"被动维修"变为"预测维护" |
| 知识沉淀 | 将老工艺师/班组长经验沉淀为可复用的 AI 知识 |

---

## 2. 适用场景分析

基于当前系统数据现状，筛选出**数据就绪度高、业务价值大、技术风险低**的三个优先场景：

### 2.1 场景矩阵

| 场景 | 数据就绪 | 业务价值 | 技术难度 | 落地周期 | 优先级 |
|------|---------|---------|---------|---------|--------|
| 质量异常智能预警 | 高（已有报工+质检数据） | 高 | 低（规则引擎） | 2-3天 | P1 |
| 智能排程建议 | 高（已有工单+排产历史） | 高 | 中（规则+统计） | 1周 | P2 |
| 设备预测性维护 | 中（OEE 数据需积累） | 高 | 中（趋势分析） | 1-2周 | P3 |
| 自然语言知识库 | 低（需人工录入工艺文档） | 中 | 高（LLM） | 2-3周 | P4 |
| 视觉质检 | 低（需产线部署相机） | 中 | 高（CV） | 待定 | P5 |

> **建议**：先落地 P1-P3，P1（质量预警）可2天内出效果，P2（排程）1周出 MVP。

### 3.2 与现有架构的集成关系

| 现有模块 | AI 调用点 | 数据流向 |
|----------|-----------|----------|
| WorkReportService | 报工完成后 → 触发质量分析 | AI 读取报工数据，返回预警 |
| SchedulingService | 排程时 → 请求 AI 建议 | AI 返回推荐产线+理由 |
| EquipmentService | 定时任务 → 读取 OEE 趋势 | AI 返回故障预测 |
| DashboardService | 预警结果 → 推送到看板 | AI → Andon / 看板 |

## 4. 场景详细设计

### 4.1 场景一：质量异常智能预警（P1）

#### 触发条件（可配置）

| 预警规则 | 阈值 | 触发动作 |
|----------|------|---------|
| 产线连续不良率飙升 | 连续 3 个工单不良率 > 5% | 推送到 Andon + 通知班组长 |
| 物料批次异常 | 某批次物料在多个工单中不良率 > 3% | 标记该批次为"重点监控" |
| 工位异常 | 某工位连续 5 次报工返工 | 推送工艺检查提醒 |

#### 技术路线

```
WorkReport 完成
    → Kafka 事件（或轮询）
    → AiQualityService.Analyze()
        ① 查询最近 N 条同产线工单
        ② 统计不良率趋势
        ③ 与历史同期对比
        ④ 命中规则 → 生成预警 DTO
    → SignalR 推送前端
    → 前端弹出预警卡片
```

> **无需 LLM**，纯后端规则引擎即可。如需自然语言解释，可选 LLM 辅助生成文案。

---

### 4.2 场景二：智能排程建议（P2）

#### 功能描述

对新建的待排程工单，AI 分析以下因素后给出建议：
- 物料对应的工艺路线复杂度和历史实际耗时
- 各产线当前负荷（已排程未完工工单）
- 交期紧急程度
- 历史同类工单在各产线的平均完成时间

#### 输出示例

```json
{
  "recommendations": [
    {
      "lineId": 2,
      "lineName": "SMT-LINE-02",
      "confidence": 0.87,
      "suggestedStart": "2026-06-07T08:00:00",
      "suggestedEnd": "2026-06-07T16:30:00",
      "reason": "该产线当前负荷仅 45%，历史同类工单平均完成时间 7.5h，可满足交期"
    }
  ]
}
```

> **第一阶段**：规则引擎+统计分析（无需 LLM）
> **第二阶段**：可选接入 LLM 生成更自然的解释文案

---

### 4.3 场景三：设备预测性维护（P3）

#### 功能描述

基于设备 OEE 历史数据，预测未来故障风险：
- **数据采集**：定时读取 Equipment OEE 指标（可用率、性能、质量）
- **趋势分析**：计算 7 日/30 日移动平均，检测下降趋势
- **预测输出**：
  - 健康度评分（0-100）
  - 预测故障窗口（"未来 3-5 天内风险较高"）
  - 建议维护动作

#### 预警等级

| 健康度 | 等级 | 动作 |
|--------|------|------|
| 90-100 | 优 | 正常监控 |
| 70-89  | 良 | 关注趋势 |
| 50-69  | 预警 | 提前安排保养 |
| < 50   | 危险 | 立即停机检修 |

## 5. 技术选型

### 5.1 后端技术

| 组件 | 选型 | 理由 |
|------|------|------|
| AI 服务层 | 纯 .NET 10（无外部框架） | 减少依赖，便于维护 |
| 统计/规则 | 手写（MathNet 可选） | 轻量，无需引入 ML.NET |
| LLM 接入（可选） | Microsoft.AI.Chat + HttpClient | .NET 原生，支持多模型 |
| 本地 LLM（可选） | Ollama Docker 容器 | 完全内网，数据不出厂 |

### 5.2 前端技术

| 组件 | 选型 | 理由 |
|------|------|------|
| AI 助手组件 | 自研 Vue 3 组件 | 复用现有 Element Plus |
| 流式输出 | Server-Sent Events (SSE) | 比 WebSocket 更轻量 |
| 图表 | 复用现有 echarts | 统一技术栈 |

### 5.3 基础设施

| 组件 | 选型 |
|------|------|
| 事件流 | 复用现有 RabbitMQ（或定时任务） |
| 缓存 | 复用现有 Redis |
| 数据存储 | 复用现有 PostgreSQL（无需向量库） |

## 6. 数据安全与隐私

### 6.1 工厂核心关切

| 关切 | 解决方案 |
|------|---------|
| 生产数据不能上外网 | P1-P3 场景全部本地规则实现，零联网 |
| LLM 对话数据泄露风险 | 可选本地 Ollama，或明确告知用户"联网模式" |
| 数据权限隔离 | 复用现有 JWT + 角色权限 |

### 6.2 部署模式

| 模式 | 网络 | 适用场景 |
|------|------|---------|
| **纯内网模式** | 不连接互联网 | 质量预警、排程建议、设备预测 |
| **增强模式**（可选） | 仅 LLM 对话走 API | 自然语言知识库问答 |

## 8. ROI 预期

| 场景 | 当前痛点 | 预期改善 | 量化指标 |
|------|---------|---------|---------|
| 质量预警 | 发现质量问题时已产生废品 | 提前 2-4 小时预警 | 废品率降低 15-20% |
| 智能排程 | 调度员经验依赖，负载不均 | 排程效率提升 | 产线利用率提升 10% |
| 设备预测 | 突发故障导致产线停机 | 计划性维护 | 非计划停机减少 30% |
